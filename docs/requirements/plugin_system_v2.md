# 插件系统 V2 显示逻辑需求文档

## 1. 背景与目标

插件系统 V1 已建立插件运行时、独立进程隔离、Manifest、Tool / View / Agent 注册、设置页入口与聊天接入等基础能力。随着插件能力逐步接入，当前显示层存在两个明显问题：

- 插件在聊天输入框中的呈现方式偏黑盒，用户只能看到一个聚合插件项，只有 hover 时才能临时查看内部能力。
- 插件在聊天中的 View 展示能力缺少统一的宿主交互框架，右侧面板目前更接近“临时展示区域”，缺少一致的头部、状态和操作规则。

V2 的目标是对插件系统的显示层进行一次聚焦升级，使插件在聊天界面中具备更清晰的可发现性、可理解性和可操作性。

本版本仅覆盖以下两块内容：

- 聊天右侧插件面板的统一头部、关闭、刷新与状态交互
- 输入框插件项从纯 hover 能力提示升级为“可展开能力预览”

V2 不重写 V1 已定义的插件运行时、隔离、Manifest、Hook、Agent、通信协议与安装启用流程。

## 2. 当前问题概述

### 2.1 输入框插件项问题

当前聊天输入框中的插件项主要存在以下问题：

- 插件在工具菜单中以聚合项显示，但能力详情依赖 hover 才能看到。
- hover 交互只适合桌面端鼠标，不适合移动端或触控设备。
- 用户选中插件后，无法稳定查看该插件包含哪些 `use_tool`、`view_tool` 或 `agent`。
- 插件描述通常较抽象，无法帮助用户快速理解“这个插件当前能做什么”。

### 2.2 聊天右侧插件面板问题

当前插件 View 在聊天中的显示能力主要存在以下问题：

- 插件右侧视图缺少统一头部，不同插件后续容易各自实现不同结构。
- 面板关闭、刷新、错误提示、空状态等交互缺少统一规范。
- 用户关闭右侧面板后，当前聊天中缺少稳定的“重新打开该视图”入口。
- 右侧面板的数据展示与插件工具调用之间缺少明确的宿主状态管理规则。

## 3. 聊天右侧插件面板 V2

### 3.1 设计原则

聊天右侧插件面板必须采用“宿主统一外壳 + 插件内容区”的结构：

- 宿主负责统一头部、状态、刷新、关闭与布局行为。
- 插件只负责返回视图数据，不负责定义宿主控制按钮和整体壳结构。
- 所有进入右侧区域的插件视图在交互上必须保持一致。

### 3.2 打开规则

当插件返回满足右侧展示条件的视图结果时，宿主应在聊天界面右侧打开插件面板。

最低判断条件：

- 返回结果中 `region=chat_side_panel`
- 返回结果中存在合法 `viewId`
- 返回结果中至少包含 `title` 或 `data`

同一时刻聊天右侧仅允许展示一个插件视图：

- 当用户或模型打开新的插件视图时，右侧面板替换当前视图。
- 不允许在右侧同时堆叠多个插件视图。
- 如果同一插件连续打开同一视图，应视为内容刷新，而不是新建第二个面板。

### 3.3 统一头部

右侧插件面板顶部必须由宿主提供统一头部。头部结构如下：

- 左侧主信息：
  - 插件名称
  - 视图标题
- 次级信息：
  - `viewId`
  - 最近刷新状态或最近执行时间
- 右侧操作区：
  - `刷新`
  - `关闭`

要求：

- 插件名称和视图标题必须是稳定可见信息，不应隐藏在 hover 中。
- `viewId` 用于调试和问题排查，显示样式应弱于主标题。
- 最近刷新状态或最近执行时间应帮助用户判断当前内容是否为最新结果。

### 3.4 刷新行为

右侧面板中的 `刷新` 为宿主统一能力，不能仅等价于“重绘当前 data”。

刷新要求：

- 刷新应重新执行最近一次与该视图关联的数据获取流程。
- 如果最近一次视图来自 `view_tool`，则优先重新执行对应视图流程。
- 如果最近一次视图由某个 `use_tool` 的结果推导而来，宿主应重新执行与之绑定的数据获取流程。
- 刷新过程中，面板进入 `loading` 状态。
- 刷新成功后更新视图内容和最近刷新时间。
- 刷新失败时，保留上一版成功内容，同时在头部或内容区展示错误状态。

V2 不要求在文档中定义底层刷新协议，只要求宿主保存最近一次可刷新的视图上下文，以支持刷新。

### 3.5 关闭行为

右侧面板中的 `关闭` 为宿主统一能力。

关闭要求：

- `关闭` 仅隐藏当前右侧面板。
- `关闭` 不影响插件启用状态。
- `关闭` 不删除聊天记录。
- `关闭` 不删除工具执行结果。

面板关闭后，聊天消息区必须保留“重新打开该视图”的入口。

该入口要求：

- 与产生该视图的消息或工具结果关联。
- 用户无需重新提问即可重新打开最近一次已关闭的面板。
- 重新打开后恢复该视图的最近结果，必要时可由用户手动点击刷新。

### 3.6 状态定义

右侧插件面板至少支持以下五种状态：

#### `loading`

- 触发时机：首次打开、手动刷新、插件正在重新获取数据
- 表现：头部或内容区显示统一加载态

#### `ready`

- 触发时机：插件视图数据正常返回
- 表现：展示插件内容区

#### `empty`

- 触发时机：插件视图成功返回，但内容为空
- 表现：展示统一空状态，例如“暂无邮件”

#### `error`

- 触发时机：插件视图获取或刷新失败
- 表现：显示统一错误提示和重试入口
- 错误状态不应强制清空上一版成功内容

#### `stale`

- 触发时机：当前展示的是旧结果，最近刷新失败或当前结果未刷新
- 表现：展示次级提示，例如“显示的是上一次成功结果”

### 3.7 内容区职责边界

插件右侧面板内容区只负责展示插件数据，不负责宿主级操作。

要求：

- 插件内容区可以是结构化列表、详情、表格、卡片或通用 JSON 兜底视图。
- 宿主不要求所有插件使用同一种内容布局。
- 但插件内容区不得覆盖宿主头部与统一操作区。

V2 推荐宿主提供通用兜底渲染能力：

- 已知 `viewId` 可使用宿主专用渲染器
- 未知 `viewId` 使用通用 JSON / key-value 视图兜底

## 4. 输入框插件项能力预览 V2

### 4.1 设计原则

输入框中的插件项应从“hover 临时提示”升级为“可展开能力预览”。

设计目标：

- 让用户在选择插件前知道该插件大致能做什么
- 让能力详情在桌面端和移动端都可访问
- 让插件项既保持紧凑，又不过度黑盒

### 4.2 默认展示结构

插件项在未展开时至少展示以下信息：

- 插件名
- 插件标签，例如“插件”
- 简短摘要
- 当前选中状态
- 展开按钮

其中简短摘要应优先帮助用户理解用途，而不是重复长描述。

宿主可以基于能力生成摘要，例如：

- “查邮件、发邮件、显示邮件列表”
- “查看项目、创建任务、展示任务面板”

宿主生成的简短摘要不要求严格依赖插件原始描述文本。

### 4.3 展开交互

插件项右侧应提供展开按钮，点击后展开能力预览区。

交互要求：

- 点击整行：选中或取消选中插件
- 点击展开按钮：仅展开或收起能力预览，不改变选中状态
- 展开状态在当前工具菜单打开期间保持
- 展开状态不因鼠标移出而立即消失
- 移动端与桌面端采用相同交互模型，不依赖 hover 才能查看能力详情

### 4.4 能力预览内容

展开后的能力预览区按能力类型分组展示：

- `use_tool`
- `view_tool`
- `agent`

每项能力应展示：

- 能力名称
- 一句短描述
- 类型标签

如果某组能力为空，可不显示该分组标题。

### 4.5 插件类型差异化展示

不同插件类型在能力预览中的信息重点不同：

#### `general_plugin`

默认强调：

- 可调用工具
- 可展示视图

摘要文案优先围绕“工具 / 视图能力”生成。

#### `agent_plugin`

默认强调：

- 可委派 Agent 能力
- Agent 可提供的视图能力

摘要文案优先围绕“可委派的智能体能力”生成，不突出内部工具。

### 4.6 长列表处理

当插件能力较多时，能力预览区必须避免撑爆工具菜单。

要求：

- 预览区可内部滚动
- 菜单整体高度需要有稳定上限
- 能力较多时可显示聚合摘要，例如“共 8 项能力”
- 不要求默认全部展开到完整高度

## 5. 状态与交互规则

### 5.1 右侧面板与聊天消息关系

聊天消息区与右侧插件面板的职责应分离：

- 聊天消息区负责记录“插件做了什么”
- 右侧插件面板负责展示“插件产出了什么结构化结果”

当插件打开右侧面板时，聊天消息区应保留简洁反馈，例如：

- “已展示邮件列表”
- “已打开任务详情面板”

不建议在主聊天流中重复渲染整块复杂结构化内容。

### 5.2 输入框与聊天面板关系

输入框插件项负责“能力发现”，右侧插件面板负责“结果展示”。

要求：

- 用户在输入框中展开插件项时，应清楚知道插件可以做什么
- 用户在聊天中触发插件后，应清楚知道结果展示在哪里
- 当右侧面板已打开时，聊天界面不应让用户误以为插件结果只存在于工具结果文本中

### 5.3 一致性要求

V2 的所有插件显示相关 UI 必须满足以下一致性原则：

- 宿主级操作位置一致
- 桌面端与移动端能力可见性一致
- 状态表达一致
- 插件间展示框架一致

V2 不要求所有插件内容长得一样，但要求所有插件“外壳逻辑”一致。

## 6. 最小接口约束

V2 对显示层依赖的最小接口约束如下：

### 6.1 右侧面板视图载荷

宿主消费的插件视图载荷至少包含以下字段：

- `region`
- `viewId`
- `title`
- `data`

要求：

- `region=chat_side_panel` 作为右侧面板渲染入口
- `viewId` 用于宿主路由具体渲染器
- `title` 用于右侧面板头部显示
- `data` 为插件内容区渲染数据

V2 不引入新的插件协议版本，只要求宿主在现有视图结果基础上统一消费。

### 6.2 可刷新视图上下文

宿主需要保存最近一次可刷新的插件视图上下文，用于支持：

- 头部 `刷新`
- 面板关闭后的重新打开
- 异常恢复后的重试

V2 不要求规定上下文存储位置，只要求行为可实现。

### 6.3 输入框能力预览数据

输入框插件项能力预览至少需要读取以下聚合能力：

- `use_tools`
- `view_tools`
- `agents`

宿主不需要在输入框中暴露插件内部实现细节，只需展示用户可感知的能力摘要和能力列表。

## 7. 验收标准

满足以下条件时，可认为 V2 插件显示逻辑符合要求：

### 7.1 输入框能力预览

- 用户在聊天输入框中可以展开插件项查看能力预览
- 桌面端和移动端都能查看插件能力详情
- 插件能力预览不依赖 hover 才可访问
- 展开后可区分 `use_tool`、`view_tool`、`agent`
- `agent_plugin` 与 `general_plugin` 的摘要和分组重点符合类型差异

### 7.2 右侧插件面板

- 插件返回 `chat_side_panel` 视图后，右侧面板显示统一头部
- 头部包含插件名、视图标题、次级信息、刷新、关闭
- 面板可关闭，关闭后聊天消息中仍可重新打开对应视图
- 点击刷新会重新执行最近一次视图获取流程，而不是仅重绘旧数据
- 加载中、空结果、错误、旧数据提示都能在头部或内容区正确呈现
- 连续打开两个插件视图时，右侧仅保留最新一个，不发生叠加

## 8. 默认假设

- `plugin_system_v2.md` 是一份聚焦插件显示优化的 V2 文档，不是完整插件系统 V2 总规范
- V1 中已定义的插件运行时、隔离、Manifest、Hook、Tool / Agent 基础机制继续沿用
- 本文档只整理需求，不包含代码实现细节
- 默认以现有聊天页、插件设置页、输入框工具菜单为落地点，不新增独立插件中心页面
- 默认由宿主统一控制右侧插件面板外壳，插件只提供内容数据
