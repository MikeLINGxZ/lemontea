# 插件系统 V1 需求文档

## 1. 背景与目标

Lemon Tea Desktop 需要提供一套基于 Node.js 的插件系统，使第三方能力可以以独立进程接入主程序，并在不影响主程序稳定性的前提下扩展聊天、工具调用、Agent、View、后台任务和通知能力。

V1 的目标是建立可运行、可管理、可隔离、可演进的插件基础设施：

- 主程序内置 Node.js 运行环境，插件统一运行在该环境之上。
- 插件以独立进程运行，插件崩溃、超时、卡死不得导致主程序崩溃。
- 插件能力通过 Manifest 和运行时注册暴露给主程序。
- 插件可分为 `agent_plugin` 和 `general_plugin` 两类，二者能力边界清晰。
- 插件可提供 View、Tool、Agent、LLM Hook、后台任务、通知等能力。
- 设置页需要提供插件管理入口，聊天输入框需要展示启用插件提供的能力。

V1 仅定义插件系统产品与技术需求，不要求在本文档中实现具体代码。

## 2. 术语定义

| 术语 | 含义 |
| --- | --- |
| Host | Lemon Tea Desktop 主程序，负责插件发现、安装、启动、通信、权限控制和 UI 集成。 |
| Plugin | 运行在 Node.js 环境中的插件包。 |
| Plugin Runtime | 主程序内置的 Node.js 运行时及插件执行所需基础文件。 |
| Plugin Process | 每个插件独立启动的 Node.js 子进程。 |
| Manifest | 插件声明文件，描述插件元信息、类型、入口、权限和静态能力。 |
| Capability | 插件向 Host 暴露的能力，包括 Tool、View、Agent、Hook、后台任务等。 |
| use_tool | 与现有大模型工具一致，可被 LLM 调用并返回结构化结果。 |
| view_tool | 被调用后可在聊天界面显示插件 View 的工具。 |
| View | 插件提供的可渲染界面，可用于设置页、聊天侧边栏或工具结果展示。 |
| A2A | Agent-to-Agent 调用协议，`agent_plugin` 内部 Agent 通过该协议与主程序 Agent 系统协作。 |

## 3. 插件类型

插件必须在 Manifest 中声明唯一插件类型。V1 支持两类插件：`agent_plugin` 与 `general_plugin`。

### 3.1 agent_plugin

`agent_plugin` 用于扩展 Agent 能力。

要求：

- 仅允许注册 Agent 和 View。
- 不允许向主程序暴露普通 Tool，包括 `use_tool` 和 `view_tool`。
- 插件内部 Agent 可以使用插件自带工具。
- 插件内部 Agent 可以使用主程序允许开放的系统内置工具。
- 插件内部 Agent 使用 A2A 与主程序交互。
- 主程序对外只展示该插件注册的 Agent 能力，不展示插件内部工具和内部执行步骤。
- 在聊天详情页中，插件 Agent 执行任务时仅显示“该 Agent 正在执行”及最终结果或错误，不显示类似内置 Agent 的过程追踪。

### 3.2 general_plugin

`general_plugin` 用于扩展通用工具和界面能力。

要求：

- 仅允许注册 Tool 和 View。
- 不允许注册 Agent。
- 可注册 `use_tool`，用于让 LLM 获取数据或执行动作。
- 可注册 `view_tool`，用于根据上下文或 `use_tool` 结果渲染界面。
- 可提供插件设置 View。

## 4. 目录与运行时

### 4.1 Node.js 运行时目录

主程序需要内置 Node.js 环境。运行时目录必须位于：

```text
<utils.GetDataPath()>/plugin_runtime
```

其中 `<utils.GetDataPath()>` 为 `backend/utils/dir.go` 中 `GetDataPath()` 返回的目录。

要求：

- 首次启动或插件系统首次初始化时，主程序需要确保 `plugin_runtime` 目录存在。
- `plugin_runtime` 下应包含当前平台可执行的 Node.js 运行环境。
- 主程序应能校验 Node.js 运行时是否存在、是否可执行、版本是否满足插件系统最低要求。
- Node.js 运行时缺失或损坏时，插件系统应进入不可用状态，并在设置页展示明确错误，不影响主程序其他功能。

### 4.2 插件安装目录

建议 V1 使用如下目录结构：

```text
<utils.GetDataPath()>/plugins
  installed/
    <plugin_id>/
      plugin.json
      package.json
      dist/
      assets/
      data/
  logs/
    <plugin_id>.log
  state/
    plugins.json
```

要求：

- `installed/<plugin_id>` 保存插件包内容。
- `data/` 为插件私有数据目录。
- `logs/` 保存插件运行日志。
- `state/plugins.json` 或等价持久化存储保存插件安装、启用、禁用、版本和权限状态。
- 插件只能默认访问自己的插件目录和数据目录，访问其他目录必须经过 Host 权限控制。

## 5. 插件生命周期

### 5.1 安装

用户在插件设置页点击“添加”后，主程序应允许选择插件包或插件目录。

安装要求：

- 校验 Manifest 是否存在且格式合法。
- 校验插件 ID、名称、版本、类型、入口文件和最低 Host 版本。
- 校验插件类型与声明能力是否匹配。
- 为插件创建私有数据目录和日志文件。
- 安装成功后插件默认处于禁用状态，除非后续产品策略明确允许安装后自动启用。

### 5.2 启用

用户启用插件时：

- Host 启动该插件独立进程。
- Host 与插件建立通信通道并完成握手。
- Host 拉取或接收插件注册能力。
- Host 将插件能力合并到工具选择、Agent 注册、Hook 链和 View 注册表。
- 启用状态持久化。

启用失败时：

- 插件状态应保持禁用或标记为启动失败。
- 设置页展示失败原因。
- 不应影响其他插件和主程序功能。

### 5.3 禁用

用户禁用插件时：

- Host 从工具选择、Agent 注册、Hook 链、View 注册表中移除该插件能力。
- Host 通知插件执行关闭流程。
- Host 在超时后强制终止插件进程。
- 禁用状态持久化。
- 正在执行的插件 Tool、Agent、后台任务应被取消，并向调用方返回明确的取消错误。

### 5.4 删除

用户删除插件时：

- 若插件正在启用，必须先禁用并停止进程。
- 删除插件安装目录。
- 删除插件能力注册记录。
- 删除插件状态记录。
- 插件私有数据默认随插件删除；如果未来支持保留数据，需要在删除确认弹窗中提供选择。V1 默认删除。

### 5.5 异常恢复

要求：

- 插件进程崩溃时，Host 应标记插件异常并清理已注册运行时能力。
- V1 可支持有限次数自动重启，默认最多 1 次；再次失败则保持异常状态。
- 插件卡死或 RPC 超时不得阻塞主程序 UI 和聊天流程。
- 插件异常需要写入日志，并在设置页插件详情中展示最近错误。

## 6. Manifest 规范

插件根目录必须包含 `plugin.json`。

建议结构：

```json
{
  "id": "com.example.mail",
  "name": "Mail",
  "version": "1.0.0",
  "description": "邮件插件",
  "type": "general_plugin",
  "main": "dist/main.js",
  "minHostVersion": "1.0.0",
  "author": "Example",
  "permissions": [
    "model.call",
    "notification.send",
    "background.task"
  ],
  "views": [
    {
      "id": "mail_list",
      "name": "邮件列表",
      "entry": "dist/views/mail-list.html"
    }
  ],
  "settingsView": {
    "id": "settings",
    "entry": "dist/views/settings.html"
  }
}
```

字段要求：

- `id`：全局唯一，建议反向域名格式；安装后不可变。
- `name`：展示名称，用于设置页和工具选择。
- `version`：语义化版本。
- `description`：插件描述。
- `type`：必须为 `agent_plugin` 或 `general_plugin`。
- `main`：Node.js 入口文件。
- `minHostVersion`：最低 Host 版本。
- `permissions`：插件申请的 Host API 权限。
- `views`：静态声明的 View，可为空。
- `settingsView`：插件设置页面声明；不存在时设置页不显示“设置”按钮。

Host 必须拒绝以下 Manifest：

- 缺少必填字段。
- 插件类型未知。
- `agent_plugin` 声明普通 Tool。
- `general_plugin` 声明 Agent。
- 入口文件不存在。
- 插件 ID 与已安装插件冲突但版本升级流程未触发。

## 7. 进程模型与隔离

### 7.1 独立进程

要求：

- 每个启用插件运行在独立 Node.js 子进程中。
- 不同插件不得共享同一个进程。
- 插件进程由 Host 管理生命周期。
- 插件进程退出、崩溃、卡死时，Host 需要隔离影响并释放资源。

### 7.2 通信协议

Host 与插件建议使用 JSON-RPC 风格协议。

协议要求：

- 每个请求包含 `id`、`method`、`params`、`protocolVersion`。
- 每个响应包含 `id`、`result` 或 `error`。
- 错误对象包含 `code`、`message`、`data`。
- 支持请求超时。
- 支持取消正在执行的请求。
- 支持 Host 主动推送事件给插件。
- 支持插件主动调用 Host API。

示例：

```json
{
  "id": "req_001",
  "protocolVersion": "1.0",
  "method": "register_use_tool",
  "params": {
    "id": "search_mail",
    "name": "搜索邮件",
    "description": "根据关键词搜索邮件",
    "inputSchema": {
      "type": "object",
      "properties": {
        "query": { "type": "string" }
      },
      "required": ["query"]
    }
  }
}
```

### 7.3 资源限制

V1 应至少定义并实现以下限制策略：

- 单次 Tool 调用默认超时 2 分钟。
- Hook 调用默认超时 10 秒。
- 插件启动握手默认超时 15 秒。
- 插件关闭默认等待 5 秒，超时强制终止。
- 插件日志需要限制大小并滚动，避免无限增长。

## 8. Host API 能力

插件只能调用已授权的 Host API。Host API 必须校验插件身份、插件状态、权限和请求参数。

### 8.1 LLM 消息发送前 Hook

接口：`before_llm_send`

触发时机：

- 用户消息组装完成后。
- 实际调用模型前。

能力：

- 读取当前聊天上下文摘要。
- 读取即将发送给模型的消息。
- 可返回修改后的消息、附加系统提示或元数据。

要求：

- 多个插件 Hook 按启用顺序或明确优先级顺序执行。
- 单个 Hook 超时或失败时，不应阻断主流程；默认记录错误并跳过该 Hook。
- 如果未来支持阻断发送，必须由权限和用户设置显式开启。V1 默认不允许插件阻断发送。

### 8.2 LLM 消息发送后 Hook

接口：`after_llm_send`

触发时机：

- 模型响应完成后。
- 助手消息持久化前或持久化后由实现决定，但文档和实现必须保持一致。

能力：

- 读取用户消息、模型响应、工具调用结果摘要和任务状态。
- 可返回附加元数据、通知请求或后台任务请求。

要求：

- Hook 失败不得影响用户看到模型响应。
- Hook 执行结果需要记录到日志或可观测事件中，便于排查。

### 8.3 插件调用模型

接口：`call_model`

能力：

- 插件可请求 Host 使用当前已配置模型或指定模型进行一次 LLM 调用。
- Host 负责复用现有模型供应商配置、密钥和调用链。

要求：

- 插件不得直接读取模型供应商密钥。
- 调用必须受权限 `model.call` 控制。
- 调用需要纳入主程序的速率、并发、取消和错误处理策略。
- Host 应记录插件发起的模型调用来源。

### 8.4 后台任务

接口：`register_background_task` / `schedule_task`

能力：

- 插件可注册后台任务。
- 后台任务可按一次性、间隔或事件触发执行。

要求：

- 后台任务必须绑定插件 ID。
- 插件禁用或删除后，其后台任务必须停止。
- 后台任务执行失败应记录日志，不应弹出无意义错误打扰用户。
- 需要限制并发，避免插件后台任务占满资源。

### 8.5 通知程序

接口：`notify`

能力：

- 插件可请求 Host 展示通知。
- 通知可包含标题、正文、等级、操作按钮和关联 View。

要求：

- 调用必须受权限 `notification.send` 控制。
- Host 需要限制通知频率，避免插件刷屏。
- 通知内容需要标记来源插件。

## 9. Tool、View 与 Agent 集成

### 9.1 use_tool

`use_tool` 与现有大模型工具一致，用于被 LLM 直接调用。

要求：

- 必须包含 `id`、`name`、`description`、`inputSchema`。
- 返回结果必须可序列化。
- Tool ID 在 Host 内部应使用命名空间避免冲突，例如 `plugin:<plugin_id>:tool:<tool_id>`。
- Tool 运行失败时，应返回结构化错误，不应让插件进程异常直接泄漏到模型上下文。
- 需要接入现有工具调用追踪和审批体系；如果插件 Tool 声明需要确认，则调用前走审批流程。

### 9.2 view_tool

`view_tool` 用于触发插件 View 展示。

典型流程：

1. 用户要求查看最新邮件列表。
2. LLM 调用邮件插件的 `use_tool` 获取邮件列表数据。
3. LLM 发现插件提供 `view_tool` 可以渲染邮件列表。
4. LLM 调用 `view_tool`，传入刚刚获取的数据或引用。
5. 聊天界面右侧显示邮件列表 View。

要求：

- `view_tool` 必须绑定一个已注册 View。
- `view_tool` 输入可以包含数据本体或 Host 可解析的数据引用。
- `view_tool` 返回值应包含 View ID、展示区域、初始数据和可选标题。
- V1 展示区域至少支持聊天详情页右侧面板。
- View 渲染失败时，聊天区需要展示可理解的错误状态。

### 9.3 View

View 是插件提供的 UI 页面或 UI 片段。

要求：

- 插件设置 View 在独立新窗口中打开。
- 聊天结果 View 显示在聊天详情页右侧。
- View 与 Host 通信必须走受控桥接 API，不允许直接访问 Host 内部对象。
- View 需要标记来源插件。
- View 关闭时，Host 需要通知插件释放相关资源。

### 9.4 Agent

插件 Agent 仅由 `agent_plugin` 注册。

要求：

- Agent 必须包含 `id`、`name`、`description`、能力说明和输入约束。
- Agent ID 在 Host 内部应使用命名空间避免冲突，例如 `plugin:<plugin_id>:agent:<agent_id>`。
- Agent 调用使用 A2A 协议。
- 插件 Agent 可以使用插件内部工具，但这些工具不暴露到主程序工具选择。
- 插件 Agent 可以使用 Host 明确开放的系统内置工具。
- 插件 Agent 执行时，聊天详情页仅显示该 Agent 正在执行、执行完成、执行失败等粗粒度状态。
- 插件 Agent 的内部推理、内部工具调用、内部多 Agent 过程不进入主程序现有详细执行追踪。

## 10. 设置页需求

### 10.1 入口

设置页需要新增“插件”设置页，样式参考现有设置页面。

要求：

- 设置菜单新增插件入口。
- 页面在桌面端采用左右布局。
- 移动端按现有设置页模式适配列表与详情切换。

### 10.2 插件列表

左侧为插件列表。

要求：

- 右上方显示添加按钮。
- 列表项展示插件名称、类型、版本、启用状态。
- 支持选中插件后在右侧展示详情。
- 插件异常时列表项需要有错误状态提示。

### 10.3 插件详情

右侧为插件详情。

内容要求：

- 插件名称、ID、版本、类型、描述、作者。
- 当前状态：已启用、已禁用、启动失败、运行异常。
- 已注册能力：Tool、View、Agent、Hook、后台任务。
- 权限列表。
- 最近错误或日志摘要。

操作按钮要求：

- 右上角显示启用/禁用按钮。
- 如果插件实现了设置界面，显示“设置”按钮。
- 始终显示删除按钮。
- 删除需要二次确认。

### 10.4 插件设置窗口

点击插件详情中的“设置”按钮时：

- Host 新建窗口。
- 新窗口加载插件提供的设置 View。
- 窗口标题应包含插件名称。
- 设置 View 关闭时需要释放关联资源。
- 插件未实现设置界面时不显示设置按钮。

## 11. 聊天输入框工具选择需求

当插件启用时，其提供的 Tool 或 Agent 需要合并到输入框工具选择中。

展示要求：

- 每个启用插件在工具选择中展示为一个聚合项。
- 聚合项名称为插件名称。
- 聚合项需要使用标签标记为“插件”。
- 鼠标悬停该项时，右侧展示该插件提供的工具和 Agent。
- 对 `general_plugin`，悬停内容展示 `use_tool` 和 `view_tool`。
- 对 `agent_plugin`，悬停内容展示该插件注册的 Agent。
- 插件内部工具不得显示在 `agent_plugin` 的悬停内容中。

选择行为：

- 选择插件聚合项表示启用该插件暴露给当前聊天的全部公开能力。
- 取消选择插件聚合项表示当前聊天不使用该插件公开能力，但不等于全局禁用插件。
- 全局禁用插件后，该插件聚合项从工具选择中移除。

兼容要求：

- 内置工具和 MCP 工具保持现有展示和选择逻辑。
- 插件能力需要与现有 `selectedToolIds` 或后续等价选择模型兼容。
- 插件聚合项应使用稳定 ID，例如 `plugin:<plugin_id>`。

## 12. 安全与权限

V1 需要建立最小权限模型。

权限建议：

| 权限 | 含义 |
| --- | --- |
| `model.call` | 允许插件通过 Host 调用模型。 |
| `notification.send` | 允许插件发送通知。 |
| `background.task` | 允许插件注册后台任务。 |
| `filesystem.plugin_data` | 允许插件访问自身数据目录。 |
| `host.tool.builtin` | 允许插件 Agent 使用 Host 开放的内置工具。 |
| `hook.llm.before_send` | 允许注册发送前 Hook。 |
| `hook.llm.after_send` | 允许注册发送后 Hook。 |

要求：

- 插件只能使用 Manifest 中声明且用户允许的权限。
- 敏感权限需要在启用插件时展示给用户确认。
- 插件不得读取主程序密钥、供应商 API Key、其他插件数据。
- Host API 必须基于插件 ID 做鉴权。
- 插件 View 不得直接执行 Host 内部未授权操作。

## 13. 稳定性、日志与错误处理

稳定性要求：

- 插件任何异常不得导致主程序崩溃。
- 插件 RPC 调用必须有超时。
- 插件启动、停止、Tool 调用、Hook 调用、Agent 调用、后台任务执行都需要可取消。
- 插件系统不可用时，主程序聊天、设置、模型调用等核心能力仍需可用。

日志要求：

- 每个插件独立日志文件。
- Host 记录插件生命周期事件。
- Tool、Agent、Hook、后台任务失败需要记录插件 ID、能力 ID、错误码和错误信息。
- 设置页展示最近错误摘要，完整日志可在后续版本提供打开入口。

错误处理要求：

- 用户可理解的错误展示在 UI。
- 技术细节写入日志。
- LLM 上下文中只注入必要且安全的工具错误摘要。

## 14. 兼容与扩展

V1 需要为后续扩展保留空间：

- Manifest 包含 `minHostVersion`，未来可加入 `maxHostVersion` 或能力版本。
- RPC 包含 `protocolVersion`。
- Capability 注册需要支持版本字段。
- Tool、Agent、View ID 采用插件命名空间，避免未来跨插件冲突。
- 权限模型应允许后续增加网络、文件系统、剪贴板、系统命令等权限。

## 15. 测试与验收标准

### 15.1 运行时与目录

- 首次启动插件系统时，会创建 `<utils.GetDataPath()>/plugin_runtime`。
- Node.js 运行时存在且可执行时，插件系统状态为可用。
- Node.js 运行时缺失或损坏时，设置页显示错误，主程序其他功能正常。

### 15.2 插件管理

- 可以安装合法插件。
- 非法 Manifest 会被拒绝并展示原因。
- 可以启用、禁用、删除插件。
- 插件状态在重启后保持一致。
- 删除插件后，其能力不再出现在工具选择、Agent 注册、Hook 链或 View 注册表中。

### 15.3 隔离与稳定性

- 单个插件崩溃不会影响主程序。
- 单个插件卡死不会阻塞聊天输入、设置页和其他插件。
- 多个插件同时启用时，进程、日志、能力和状态相互隔离。
- 插件启动失败时，其他插件仍可正常运行。

### 15.4 general_plugin

- `general_plugin` 可以注册 `use_tool` 和 `view_tool`。
- LLM 可以调用 `use_tool` 并获得结构化结果。
- LLM 可以调用 `view_tool`，并在聊天详情页右侧渲染插件 View。
- `general_plugin` 不能注册 Agent；尝试注册时 Host 拒绝。

### 15.5 agent_plugin

- `agent_plugin` 可以注册 Agent 和 View。
- `agent_plugin` 不能向主程序暴露普通 Tool；尝试注册时 Host 拒绝。
- 聊天输入框中只展示插件聚合项和插件 Agent，不展示内部工具。
- 插件 Agent 执行时，聊天详情页只显示粗粒度执行状态。
- 插件 Agent 可以按权限使用 Host 开放的内置工具。

### 15.6 Hook、后台任务与通知

- `before_llm_send` 按顺序执行，失败时跳过并记录日志。
- `after_llm_send` 失败时不影响用户看到模型响应。
- 插件可以在授权后调用模型，且不能读取模型密钥。
- 插件可以注册后台任务，禁用插件后后台任务停止。
- 插件可以在授权后发送通知，通知标记来源插件。

### 15.7 UI

- 设置页存在插件入口。
- 插件设置页左侧为插件列表，右侧为插件详情。
- 插件列表右上方有添加按钮。
- 插件详情右上角有启用/禁用、设置、删除按钮。
- 未实现设置界面的插件不显示设置按钮。
- 点击设置按钮会在新窗口打开插件设置 View。
- 聊天输入框工具选择中，启用插件以插件名称显示为聚合项，并带“插件”标签。
- 鼠标悬停插件聚合项时，右侧展示该插件提供的 Tool 和 Agent。

## 16. V1 默认策略

- 插件安装后默认禁用。
- 插件删除时默认删除插件私有数据。
- Hook 失败默认跳过，不阻断 LLM 请求。
- 插件启动失败默认不自动无限重试，最多重试 1 次。
- 插件 Agent 内部过程默认不进入主程序详细执行追踪。
- 插件聚合项选择表示当前聊天启用该插件公开能力，不改变插件全局启用状态。
